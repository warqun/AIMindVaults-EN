# Claude Code RTK 토큰 절약 구조와 한계

## 핵심 요약

- RTK는 `Claude Code`가 터미널 출력 전체를 문맥에 넣으면서 발생하는 토큰 낭비를 줄이기 위해, 출력 로그를 압축해서 전달하는 도구다.
- 핵심 지점은 `PreToolUse` Hook으로 명령을 `rtk-cli` 경유로 바꾸는 구조이며, 실제 터미널 실행 자체를 없애는 것은 아니다.
- `grep`, `find`, `ls`, `git diff`, `npm test`처럼 출력량이 큰 명령에서 특히 이득이 크고, 대형 코드베이스 탐색일수록 효과가 커진다.
- 다만 압축은 정보 손실을 동반하므로, 세부 로그가 중요한 상황에서는 후속 확인 명령이 늘어나 오히려 토큰 절감 효과가 줄어들 수 있다.
- 영상의 결론은 RTK가 기존 절약 습관을 대체하는 만능 해법이 아니라, CLI 출력 낭비를 줄여주는 보조 장치라는 점이다.

## RTK가 줄이려는 문제

영상의 문제의식은 단순하다. `Claude Code`가 코드를 짜는 동안 실제로는 터미널을 많이 사용하고, 그 과정에서 나온 시스템 출력도 다시 모델 입력으로 들어간다. 그런데 이 출력 중 상당수는 같은 경로 반복, 장황한 성공 로그, 중복된 파일 목록처럼 모델이 그대로 읽지 않아도 되는 정보다.

RTK는 바로 이 지점을 겨냥한다. "실행 결과는 유지하되, 모델에게는 필요한 정보만 압축해서 전달하면 토큰을 크게 줄일 수 있지 않나"라는 발상이다. 영상에서는 이 절감 폭을 대략 `60~90%` 수준으로 소개하지만, 이는 일반 보장값이라기보다 도구의 목표와 대표 사례에 가깝게 이해하는 편이 안전하다.

## 영상에서 보여준 압축 사례

### 1. 파일 탐색 계열 명령

- `grep`, `find`, `ls`처럼 코드베이스를 훑는 명령은 반복 경로와 구조 정보가 많다.
- RTK는 이런 출력에서 묶을 수 있는 항목을 그룹화해서 더 짧은 형태로 보여준다.
- 소규모 프로젝트에서는 체감이 약해 보여도, 파일 수가 매우 많은 코드베이스에서는 절감 폭이 크게 커질 수 있다는 설명이다.

### 2. 변경 확인 계열 명령

- 영상에서는 `git diff`도 지원 예시로 든다.
- 수정 후 상태를 확인하려고 자주 보는 출력이지만, 그대로 전달하면 길이가 금방 커진다.
- RTK는 이 역시 더 짧은 표현으로 바꿔 모델 문맥 부담을 줄이려 한다.

### 3. 테스트 러너 계열 명령

- `npm test`처럼 로그가 긴 명령은 성공했을 때 특히 낭비가 심하다.
- 영상 설명 기준으로 RTK는 성공 케이스에서는 핵심 성공 결과 위주로만 전달한다.
- 반대로 실패 케이스에서는 모델이 수정해야 하므로 에러 로그를 포함해 보여준다.

핵심은 "모든 로그를 무조건 줄인다"가 아니라, 수정 판단에 필요한 정보는 남기고 장황한 부분을 먼저 걷어내는 데 있다.

## RTK 작동 구조

영상에서 설명한 구성 요소는 두 개다.

1. `rtk-cli`
2. `rtk-rewrite.sh`

`rtk-cli`는 실제 출력 압축을 담당하는 실행 파일이고, `rtk-rewrite.sh`는 `Claude Code`가 실행하려는 명령을 RTK 경유 명령으로 바꾸는 Hook 역할을 한다. 영상에서는 이 지점을 `PreToolUse`에 연결된 커스텀 Hook으로 설명한다.

흐름을 단순화하면 다음과 같다.

`Claude Code → PreToolUse Hook → rtk-rewrite.sh → rtk-cli → 실제 터미널 실행 → 압축된 결과를 Claude Code에 전달`

즉, RTK는 터미널을 대체하는 것이 아니라 터미널과 모델 사이에 끼어드는 중간 계층이다. 실제 명령은 계속 실행되지만, 모델이 읽는 출력 형식만 바뀐다.

## 효과가 큰 조건

- 파일 수가 많고 탐색 명령이 자주 나오는 대형 프로젝트
- 테스트 로그나 빌드 로그가 긴 환경
- `grep`, `find`, `ls`, `git diff` 같은 출력량 많은 명령을 반복하는 워크플로

반대로 효과가 약할 수 있는 조건도 영상에서 같이 짚는다.

- 소규모 프로젝트처럼 명령 출력 자체가 짧은 경우
- CLI 탐색보다 편집과 사고 비중이 더 큰 작업
- 상세 로그를 그대로 봐야 다음 판단이 가능한 상황

결국 RTK는 "모든 작업에서 일률적으로 60% 절감" 도구가 아니라, **CLI 출력 비중이 큰 작업에서 특히 강한 최적화**로 보는 편이 맞다.

## 영상의 검증 방식과 사용 팁

영상에서는 `SWE-Bench` 계열 문제를 활용해 RTK 사용 전후를 비교했다고 설명한다. 화자는 한 문제를 여러 조건으로 `8번` 실행해 결과를 봤고, 전반적으로 긍정적인 결과가 나왔다고 말한다.

다만 자막만으로는 화면에 표시된 정확한 수치 표를 모두 복원하기 어렵다. 그래서 이 노트에서는 "반복 테스트에서 긍정적 경향이 나왔다"는 수준까지만 확인된 사실로 남긴다.

실사용 팁으로는 `RTK Gain` 명령으로 자신의 프로젝트에서 실제 절감량을 정량 확인하라고 권한다. 이 부분은 중요하다. RTK는 프로젝트 구조에 따라 효과 편차가 크기 때문에, 홍보 문구보다 **내 코드베이스 기준 측정값**을 먼저 보는 편이 낫다.

## 한계와 Trade-Off

영상 후반의 핵심 경고는 `토큰 절약`과 `정확성` 사이의 trade-off다.

- 출력을 압축하면 정보 손실 가능성이 생긴다.
- 정보가 너무 줄어들면 모델이 세부 사항을 다시 확인하려고 추가 명령을 더 실행할 수 있다.
- 그러면 절감한 토큰이 후속 탐색 비용으로 다시 나갈 수 있다.

화자는 초기 GitHub 이슈에서 이런 우려가 있었음을 언급하지만, 본인 사용에서는 큰 문제를 못 봤다고 말한다. 동시에 "발생 확률이 0%는 아니므로 알고 써야 한다"고 정리한다.

인상적인 부분은 개발 방향에 대한 해석이다. 영상에서는 최근 업데이트 흐름이 과도한 압축보다 실제 출력과 더 가까운 형태를 유지하는 쪽으로 가고 있다고 본다. 즉, 과장된 절감보다 안정성과 견고함을 우선하는 방향으로 이해하고 있다.

## 이 영상에서 가져갈 실전 판단

1. RTK는 `Claude Code` 전체 사용량을 마법처럼 줄여주는 도구가 아니라, **CLI 출력 낭비를 겨냥한 특화 최적화**다.
2. 따라서 기존의 대화 관리, 프롬프트 정리, 작업 범위 축소 같은 절약 습관을 대체하지 않는다.
3. CLI 탐색이 많은 리포지토리라면 도입 우선순위가 높다.
4. 상세 로그가 중요한 디버깅 구간에서는 압축 품질이 실제 생산성에 맞는지 확인해야 한다.

## 내 적용 메모

- AIMindVault 관점에서는 CLI 출력이 길어지는 `index`, `review`, `diff`, `test` 계열 흐름과의 궁합을 먼저 떠올릴 수 있다.
- 다만 현재 작업 방식은 에이전트가 이미 출력 요약을 부분적으로 수행하고 있으므로, RTK 도입 가치는 "중복 출력이 실제로 얼마나 많은가"를 측정한 뒤 판단하는 편이 맞다.
- 협업 환경 관점에서는 [[20260417_AI_에이전트_협업환경_구축법_오픈클로_영상_정리]]가 역할 분리와 검수 구조를 설명했다면, 이번 영상은 그 아래층에서 **CLI 출력 최적화**가 어떻게 비용을 줄이는지를 보여주는 보완 사례다.
