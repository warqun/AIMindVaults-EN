# AI 에이전트 협업환경 구축법과 오픈클로 영상 정리

## 핵심 요약

- 영상의 핵심은 `AI가 실수하면 AI로 한 번 더 검증한다`는 다중 검수 구조다.
- 중앙 오케스트레이터가 사용자 요청을 역할별 에이전트에게 분배하고, 승인과 반려 루프를 반복한다.
- 구현 단계만 자동화하는 것이 아니라 코드 리뷰, QA, 위키 문서화까지 같은 파이프라인에 넣는다.
- 비용 관점에서는 API 호출형보다 CLI와 계정 연동형을 활용하는 편이 개인 개발자에게 현실적이라는 관점을 제시한다.
- 완전 무인화가 아니라 사람이 최종 방향성과 결과를 확인하는 운영 모델로 이해하는 것이 맞다.

## 영상에서 설명한 구조

1. **중앙 서버**
   오픈클로라고 부르는 서버가 사용자 요청을 받고, 아래의 CLI 에이전트 집합을 제어한다.
2. **CLI 에이전트 집합**
   `Claude Code`, `Codex`, `Gemini CLI` 같은 도구를 웹 브라우저가 아니라 로컬 CLI 실행 단위로 묶는다.
3. **역할 분리**
   `Director`, `Planner`, `Developer`, `Reviewer`, `QA`, `Wiki`처럼 책임을 잘게 쪼개 각 에이전트가 좁은 문맥만 보게 만든다.
4. **반려와 재작업**
   디렉터가 기획 초안과 구현 결과를 다시 검사하고, 누락이나 방향 오류가 있으면 반려 후 재작업 루프를 돈다.
5. **문서화까지 포함한 완료 조건**
   코드가 끝났다고 바로 종료하지 않고, 마지막에 위키를 갱신해 다음 작업자가 코드 대신 문서를 보고 이어받을 수 있게 한다.

## 영상 기준 운영 루프

`사용자 요청 → 디렉터 정리 → 기획 초안 작성 → 기획 합의 → 디렉터 검수 → 개발 → 코드 리뷰 → QA → 최종 검수 → 위키 갱신`

영상에서 인상적인 점은 각 단계가 단순 직렬 처리로 끝나지 않는다는 것이다. 기획 단계 안에서도 에이전트끼리 서로 다른 시각으로 합의 과정을 반복하고, 개발 단계에서도 리뷰와 QA를 별도 역할로 분리해 같은 실수를 다시 잡는다.

## AI 에이전트 협업환경 구축법

### 1. 중앙 오케스트레이터를 먼저 둔다

- 입력 지점을 하나로 고정해야 한다.
- Discord, 웹 UI, CLI 중 하나를 사용자 진입점으로 두고, 나머지는 내부 실행기로 본다.
- 오케스트레이터는 요청 분배, 상태 추적, 승인과 반려 기록을 맡는다.

### 2. 역할을 좁게 나눈다

- `Director`: 사용자 의도와 최종 승인 기준만 본다.
- `Planner`: 요구사항을 작업 단위와 기획 문서로 바꾼다.
- `Developer`: 승인된 기획만 구현한다.
- `Reviewer`: 기획 대비 구현 일치 여부를 본다.
- `QA`: 실제 동작 증거를 만든다.
- `Wiki`: 최종 상태를 문서화한다.

핵심은 모든 에이전트에게 "전체를 다 보게 하지 않는 것"이다. 영상에서도 대형 프로젝트는 결국 더 잘게 쪼개야 한다고 반복해서 말한다.

### 3. 승인과 반려를 명시적 상태로 둔다

- 초안 작성
- 합의 완료
- 디렉터 승인
- 구현 완료
- 리뷰 통과
- QA 통과
- 문서화 완료

이 상태가 없으면 여러 에이전트가 동시에 움직여도 서로 다른 버전을 기준으로 작업하게 된다.

### 4. QA를 증거 기반으로 자동화한다

- 테스트 시트
- 스크린샷
- 수정 전후 비교
- 로그나 오류 메시지

영상에서는 UI 폭이 잘못된 부분을 수정한 뒤, 해당 부분만 캡처해서 보여주는 예시를 든다. 이 방식은 "고쳤다"는 말보다 훨씬 신뢰할 수 있다.

### 5. Harness 방식 검증을 넣는다

영상에서 말하는 핵심 개념은 "하네스 엔지니어링"이다. 구현이 끝나면 자동 검사기를 강제로 돌리고, 실패 시 다시 AI에게 넘겨 통과할 때까지 반복한다. 즉, 프롬프트만 던지는 구조가 아니라 실행과 검증이 포함된 반복 루프가 중요하다.

### 6. 문서를 산출물로 포함한다

- 위키는 부가 작업이 아니라 완료 조건에 포함해야 한다.
- 다음 작업자는 코드 전체를 다시 읽기보다 최신 위키를 기준으로 계획을 세운다.
- 문서가 최신이면 새 에이전트 투입 속도가 빨라진다.

### 7. 작업이 커지면 파이프라인 자체를 여러 개로 나눈다

- 기능 단위 분할
- 서버와 클라이언트 분할
- DB나 배치 작업 분할
- QA 전용 서브파이프라인 분할

영상 후반부에서는 한 번의 파이프라인이 8시간까지 길어질 수 있다고 말하며, 그래서 디렉터가 여러 개의 작업 흐름으로 다시 분배하는 구조를 만들고 있다고 설명한다.

## 개인 개발자 기준 최소 구성

1. `Director` 1개
2. `Builder` 1개
3. `Reviewer` 1개
4. `QA` 1개
5. `Wiki` 1개

처음부터 모든 역할을 세분화할 필요는 없다. 다만 `구현`과 `검수`를 같은 에이전트에게 맡기면 오류가 그대로 통과할 확률이 높아진다.

## 확인된 내용과 의견 구분

- 확인된 내용:
  영상은 실제 구축 중인 다중 에이전트 파이프라인을 챕터와 예시 화면으로 설명한다.
- 확인된 내용:
  역할 분리, 승인과 반려, QA 캡처, 위키 갱신, 작업 분해가 핵심 패턴으로 반복된다.
- 화자 의견:
  빅테크 감원과 프로그래머 시장 변화에 대한 해석은 화자의 의견이며, 영상 내부에서 별도 근거가 제시되지는 않는다.
- 화자 의견:
  비용 우위나 범용성도 실제 사용 도구, 요금제, 작업 규모에 따라 달라질 수 있다.
